## Changing Design Tokens (Colors, Fonts, Spacing)

Design tokens are the single source of truth for all visual values across the Print Room ecosystem. Changing a token here updates every app that consumes it.

**File to edit:** `src/tokens/design-tokens.json`

### Example Changes

**Change the primary brand color (Print Room Blue):**
```json
"brand": {
  "primary": "#2B3990",       // ← Change this hex value
  "primaryDark": "#232E73",
  "primarySoft": "#3A4BA8"
}
```

**Change a status color:**
```json
"status": {
  "success": "#10b981",       // ← Green for completed/confirmed
  "warning": "#f59e0b",       // ← Amber for attention needed
  "error": "#ef4444",         // ← Red for failures
  "info": "#3b82f6"           // ← Blue for informational
}
```

**Change the primary font:**
```json
"fontFamily": {
  "sans": "'DM Sans', system-ui, sans-serif",    // ← Main body font
  "display": "'DM Sans', system-ui, sans-serif",  // ← Heading font
  "mono": "JetBrains Mono, monospace"              // ← Code/data font
}
```

> **See all current token values** on the [Design Tokens](?path=/docs/documentation-design-tokens--docs) page — every colour, font, spacing and shadow is shown visually.

## Visual Review with Chromatic

Chromatic is the visual regression testing tool that catches unintended UI changes automatically.

### How It Works

When a developer opens a Pull Request:

1. **Chromatic builds your Storybook** in the cloud
2. **Screenshots every story** (currently ~43 stories across all components)
3. **Compares each screenshot** pixel-by-pixel against the last approved baseline
4. **Flags any visual differences** — even 1px of spacing or a slight colour shift
5. **Posts a status check** on the GitHub PR with a link to review

### Your Role as a Reviewer

When you receive a Chromatic review link:

## Editing Documentation Pages

Documentation pages (like this one) use **MDX** format — Markdown with JSX support.

**Location:** `apps/storybook/src/stories/docs/`

### Current Documentation Pages

| Page | File | Purpose |
|------|------|---------|
| Introduction | `Introduction.mdx` | Overview of the component library and tech stack |
| Design Tokens | `DesignTokens.mdx` | Visual reference for all tokens (colours, type, spacing, shadows) |
| Design Tokens Guide | `DesignTokensGuide.mdx` | How to edit tokens (for non-developers) |
| Contributing Guide | `ContributingGuide.mdx` | This page |

### Basic MDX Syntax

```mdx
# Heading 1
## Heading 2
### Heading 3

**Bold text** and *italic text*

- Bullet point
- Another point

> This is a callout / quote block

| Column 1 | Column 2 |
|----------|----------|
| Cell 1   | Cell 2   |
```

### Creating a New Documentation Page

1. Create a new `.mdx` file in `apps/storybook/src/stories/docs/`
2. Start with this template:

```mdx
import { Meta } from '@storybook/addon-docs/blocks';

<Meta title="Documentation/Your Page Name" />

# Your Page Title

Your content here...
```

3. Open a PR with the new file — it will appear in the Storybook sidebar automatically.
